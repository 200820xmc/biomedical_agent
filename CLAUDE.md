## 二、项目概述

### 2.1 项目名称

**多通道医学信号智能分析 Agent 平台**

### 2.2 项目目标

建设一个面向医学信号研究场景的智能分析平台。平台可能包括：

- 多通道医学信号导入；
- 医学信号数据管理；
- 信号质量检查；
- 信号滤波和预处理；
- 波形可视化；
- STFT 时频变换；
- Mel 或 Log-Mel 时频特征生成；
- 多通道空间信息建模；
- 多通道特征融合；
- 医学信号三分类模型；
- 模型推理；
- 模型评价；
- 医学文献知识库；
- RAG 检索增强问答；
- Agent 任务规划；
- 医学信号分析工具调用；
- 智能分析报告生成；
- 实验结果展示。

以上是项目可能涉及的能力，不代表必须一次全部开发。

实际开发范围必须以：

1. 当前代码仓库；
2. 项目需求文档；
3. README；
4. 已完成代码；
5. 用户当前提出的开发任务；

为准。不得为了展示技术复杂度，擅自扩大项目范围。

### 2.3 项目定位

本项目当前定位为：

> 面向医学信号科研和教学场景的智能分析原型平台。

当前代码仓库为 **AVF 动静脉瘘狭窄深度学习科研助手** (`avf-research-assistant` v2.0.0)，是从智能 OnCall 运维系统迁移而来。

- **核心功能**：基于 RAG 知识库的 AVF 论文检索、阅读与科研分析
- **领域**：动静脉瘘（AVF）狭窄的深度学习分类/检测
- **用户**：医学背景研究生，编程初学者，目的是帮师弟快速了解课题方向
- **界面语言**：中文为主，英文术语保留原文

不得直接将系统描述为：

- 医疗诊断系统；
- 临床诊断系统；
- 自动疾病诊断系统；
- 可以替代医生的系统。

界面、报告和文档中应明确：

> 本平台生成的分析结果仅用于科研、教学和算法验证，不构成临床诊断意见。

不得伪造：

- 临床有效性；
- 模型准确率；
- 实验结果；
- 数据规模；
- 模型性能提升；
- 医疗结论。

没有真实实验结果时，应明确写：

> 当前结果为测试结果、模拟结果或待验证结果。

## 三、开发总原则

项目开发优先级如下：

正确性
> 可理解性
> 可运行性
> 可维护性
> 可验证性
> 开发速度
> 技术复杂度

- 清晰；

## 八、领域开发规范

### 8.1 多通道医学信号

处理医学信号时，必须明确说明数据维度。例如：

```text
单条信号：[通道数, 时间采样点]
批量信号：[批次, 通道数, 时间采样点]
时频特征：[批次, 通道数, 频率维度, 时间维度]
```

不得在没有说明的情况下：调换通道顺序、调换时间维度、调换频率维度、自动删除通道、自动补充通道、改变采样率、改变信号单位。

涉及重采样、滤波、去噪、归一化、信号截断、信号补零、STFT、Mel 变换、Log-Mel 变换、数据增强等操作时，需要记录参数。

重要参数优先放入：配置文件、配置类、环境变量、实验配置文件。不得散落在多个代码文件中。

### 8.2 三分类模型

当前核心研究目标以医学信号三分类为主。具体类别名称以数据集标签和项目配置为准。

不得在代码中大量硬编码类别名称，类别信息应尽量统一配置。

训练集、验证集和测试集划分时，应优先防止同一研究对象的数据同时进入不同数据集。预处理参数不得利用测试集信息进行拟合。

模型评价根据研究任务选择：Accuracy、Precision、Recall、Macro-F1、混淆矩阵、AUROC、AUPRC。

不得只展示准确率而忽略类别不平衡问题。不得虚构评价结果。

### 8.3 Agent 开发

Agent 不等于普通聊天机器人。实现 Agent 功能前，应明确：

1. 用户提出什么任务；
2. Agent 如何理解任务；
3. Agent 可以调用哪些工具；
4. 每个工具解决什么问题；
5. 工具输入/输出是什么；
6. Agent 如何返回分析结果。

优先采用：**一个 Agent + 多个明确工具**，不要一开始构建复杂的多 Agent 系统。

Agent 工具应具有：明确名称、单一职责、清晰输入/输出、参数校验、错误处理、日志记录。

不得允许大模型直接生成未经验证的医学分析结论。涉及医学信号分析时，应优先调用确定性的信号处理工具或模型工具，再由大模型解释结果。

### 8.4 RAG 和知识库

知识库回答需要区分：文献原文、系统检索结果、大模型总结、推测性内容。

回答应尽量保留来源信息。

不得伪造：文献、作者、DOI、实验数据、医学结论。

没有检索到可靠内容时，应明确说明：**"当前知识库中未检索到足够依据。"**

不要让模型根据常识假装引用知识库。

7. 不要在无代理环境下运行（DashScope API 需要 `127.0.0.1:7890` 代理）

## 十四、面向初学者的代码讲解要求

完成修改后，不要只输出"功能已经完成"。必须解释以下内容：

### 本次修改解决了什么问题

使用生活化语言说明。

### 修改了哪些文件

```text
backend/app/services/signal_service.py

作用：负责医学信号预处理。
本次修改：新增多通道信号维度检查。
```

### 核心代码如何运行

解释：输入是什么 → 经过什么处理 → 输出是什么。

### 数据流

尽量使用流程图形式：

```text
用户上传信号
        ↓
API 接收文件
        ↓
检查文件格式
        ↓
解析多通道数据
        ↓
调用信号预处理模块
        ↓
返回处理结果
        ↓
前端显示波形
```

### 关键代码讲解

重要代码需要逐段解释。每段说明：代码做什么、为什么需要、输入是什么、输出是什么、出错可能发生在哪里。

标准导入语句和简单样板代码可以简要说明，不需要机械地逐行解释。

## 十五、开发完成后的固定汇报格式

每次完成代码开发后，按照以下结构汇报。

### 一、任务完成情况

说明：已完成什么、未完成什么、当前是否可以运行。

### 二、本次修改文件

逐个说明文件作用和修改内容。

### 三、功能运行流程

使用流程图说明：输入 → 处理 → 调用 → 输出。

### 四、关键代码讲解

解释核心代码。

### 五、运行方法

提供可以直接复制的命令，说明应该在哪个目录运行。

### 六、预期结果

说明正常情况下终端显示什么、浏览器显示什么、接口返回什么。

### 七、测试结果

说明运行了哪些测试、哪些通过、哪些没有运行。

### 八、当前问题和风险

不得隐藏问题。

### 九、面试表达

使用 3～6 句话说明：面试时如何介绍本次功能。

### 十、推荐下一步

只推荐最合理的 1～3 个后续任务。

## 十六、完成定义与禁止行为

### 完成定义

不能只因为代码已经生成，就认为任务完成。任务完成需要尽量满足：

- 功能已经实现；
- 代码能够运行；
- 核心流程已经验证；
- 没有明显导入错误；
- 没有明显语法错误；
- 没有破坏原有功能；
- 关键异常已经处理；
- 重要代码已经解释；
- 运行方法已经说明；
- 文档已经同步；
- 项目负责人能够理解本次修改。

### 禁止行为

禁止：

- 未分析项目就大规模写代码；
- 一次重写整个项目；
- 随意更换技术栈；
- 为展示能力过度设计；
- 创建大量没有使用的文件；
- 创建无法运行的空壳模块；
- 隐藏错误；
- 伪造测试结果；
- 伪造实验结果；
- 伪造模型性能；
- 伪造医学结论；
- 把模拟数据描述为真实数据；
- 直接提交密钥；
- 未经允许删除重要代码；
- 未经允许修改大量无关文件；
- 给初学者输出大量没有解释的代码。
