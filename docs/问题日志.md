## 问题 #1 — `.env` 注释符导致 Docker Compose 解析失败

**时间**: 2026-07-08

**现象**:
```
failed to read .env: line 29: unexpected character ";" in variable name "; MCP_CLS_TRANSPORT=sse"
```

**原因**:
- 原项目 `.env` 文件第 29-30 行使用了 `;` 作为注释符（INI 格式风格）
- Docker Compose 解析 `.env` 时不支持 `;` 注释，只认 `#` 注释
- 这是项目原作者留下的，在 Linux/Mac 上可能不会触发（取决于启动方式），但在 Windows Docker Desktop 中直接报错

**解决方案**:
- 将 `; MCP_CLS_TRANSPORT=sse` → `# MCP_CLS_TRANSPORT=sse`
- 将 `; MCP_CLS_URL=https://mcp-api.tencent-cloud.com/sse/xxxx` → `# MCP_CLS_URL=...`

**状态**: ✅ 已修复

## 问题 #2 — Docker 镜像拉取失败：所有注册表超时

**时间**: 2026-07-08

**现象 1 — Docker Hub 镜像源大小校验失败**:
```
failed size validation: 6956 != 1023: failed precondition
```

**现象 2 — 直连超时**:
```
Docker Hub: The operation has timed out.
quay.io: The operation has timed out.
```

**现象 3 — quay.io TLS 握手超时**（拉取 etcd 镜像时）:
```
failed to resolve reference "quay.io/coreos/etcd:v3.5.18":
net/http: TLS handshake timeout
```

**原因**:
1. **网络环境限制** — 当前网络无法直连 Docker Hub (`registry-1.docker.io`) 和 quay.io
2. **镜像源数据损坏** — `daemon.json` 中配置的部分免费镜像源缓存了不完整的镜像层，导致 `failed size validation` 错误
3. **quay.io 无镜像覆盖** — Docker `registry-mirrors` 只代理 Docker Hub，`quay.io/coreos/etcd` 走的是 quay.io 直连，镜像源帮不上忙

**影响的容器镜像**:
| 镜像 | 来源 | 是否可用镜像加速 |
|------|------|:---:|
| `quay.io/coreos/etcd:v3.5.18` | quay.io (Red Hat) | ❌ 不受 Docker Hub 镜像源影响 |
| `minio/minio:RELEASE.2023-03-20T20-16-18Z` | Docker Hub | ⚠️ 受镜像源影响但数据异常 |
| `milvusdb/milvus:v2.5.10` | Docker Hub | ⚠️ 受镜像源影响但数据异常 |
| `zilliz/attu:v2.5` | Docker Hub | ⚠️ 受镜像源影响但数据异常 |

**可能的解决方向**:

| 方案 | 描述 | 难度 |
|------|------|------|
| A | 使用可用的代理/机场/VPN 直连 | 低（需外部工具） |
| B | 更换 `daemon.json` 中可靠的镜像源 | 中（需逐个验证） |
| C | 为 quay.io 额外配置镜像代理 | 中（quay.io 国内镜像稀缺） |
| D | 将 etcd 镜像替换为 Docker Hub 上可拉取的版本（如 `bitnami/etcd`） | 中（需验证兼容性） |
| E | 找已下载好镜像的机器导出 → 离线导入 | 高（需另一台能翻的机器） |

## 问题 #6 — Docker Desktop (WSL2) 无法访问 Windows 代理

**时间**: 2026-07-08

**现象**:
- Windows 通过代理 `127.0.0.1:7890` 可以正常访问 Docker Hub
- Docker daemon（运行在 WSL2 虚拟机中）持续报 TLS 握手超时：
```
net/http: TLS handshake timeout
```

**原因**:
1. Docker Desktop 使用 WSL2 作为后端，WSL2 有独立的网络命名空间（IP 地址段 `172.28.x.x`）
2. 代理软件（Clash/V2Ray）只在 Windows 的 `127.0.0.1` 上监听，不接受来自 WSL2 的外部连接
3. WSL2 无法通过 `127.0.0.1:7890` 访问到 Windows 上的代理
4. Docker Desktop 内建的代理转发（`http.docker.internal:3128`）对 `auth.docker.io` 的 TLS 连接处理不稳定

**解决步骤**:
1. ✗ 尝试修改 `settings-store.json` 配置代理 → Docker Desktop 未正确应用
2. ✗ 尝试用 WSL2 网关 IP (`172.28.176.1`) 代替 `127.0.0.1` → 代理仍拒绝外部连接
3. ✗ 尝试 `netsh portproxy` 端口转发 → 需管理员权限
4. ✓ **最终方案**: 用户在 Docker Desktop 界面手动配置代理（Settings → Resources → Proxies）
5. ✓ 去掉 `daemon.json` 中损坏的镜像源，通过代理直连拉取

**状态**: ✅ 已解决（通过 Docker Desktop UI 手动配置代理）

3. ✓ **最终方案**: 去掉镜像源 + 代理直连 + 多次重试。Milvus 核心镜像（etcd、minio、milvus standalone）全部拉取成功；attu（网页管理界面）未拉到但不影响核心功能

## 问题 #8 — Loguru 输出 emoji 在 Windows GBK 控制台报编码错误

**时间**: 2026-07-08

**现象**:
- uvicorn 启动日志中大量 `UnicodeEncodeError: 'gbk' codec can't encode character` 错误
- 涉及 emoji 字符如 🚀📝🌐📚🔌✅

**原因**:
- `app/utils/logger.py` 中 loguru 输出到 `sys.stdout`（控制台）
- Windows 控制台默认使用 GBK 编码，无法编码 emoji（Unicode 高位字符）
- **不影响功能**，只是日志显示时 emoji 变成乱码

**解决方案**（低优先级，暂不修复）:
- 方案 A: logger 输出中移除 emoji
- 方案 B: 改用 UTF-8 编码的日志文件输出
- 方案 C: 设置 PowerShell 编码为 UTF-8: `$OutputEncoding = [Console]::OutputEncoding = [Text.Encoding]::UTF8`

**状态**: 🔵 已知但暂不修复（不影响功能）
