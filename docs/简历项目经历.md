### AVF Research Assistant——科研文献RAG Agent平台

**个人项目｜Python / FastAPI / LangChain / LangGraph / DashScope / Milvus / Ragas / Docker**

- 基于 **LangChain `create_agent`与LangGraph `MemorySaver`** 构建科研问答Agent，封装知识检索、PDF待处理列表、PDF入库和任务状态查询等本地工具，形成“需求理解—工具决策—证据检索—引用回答”的执行闭环；使用`session_id`维护进程内多轮会话，同时支持JSON和SSE流式响应。
- 完成 **Prompt与Context Engineering**：通过系统提示词约束引用格式、证据不足处理、医学因果边界和PDF状态机，防止编造引用及将相关性误写为因果；将检索结果组织为“证据编号—作者年份—章节—正文—参考文献”，结合12,000字符预算和`chunk_index`邻接扩展，为模型提供可追溯上下文。
- 设计两阶段检索链路：**Milvus Dense Top-20 → 精确Chunk去重 → `qwen3-rerank` Top-10 → 0.65阈值 → Top-5基础证据**；针对Rerank超时、网络异常、HTTP失败、空结果和不完整响应设计整批向量排序降级，并在真实Rerank全部低于阈值时保留Top-3，避免外部模型故障导致零上下文。
- 打通PDF/Markdown/TXT文献处理链路：PDF通过xParse后台解析，采用标题感知与递归字符分块，以`text-embedding-v4`生成1024维向量写入Milvus；设计`source_id、chunk_index、content_hash、chunk_id`元数据，完成 **66篇文献、1,284个Chunk** 的统一重建、书目去重和索引连续性审计。
- 构建50题人工审核的多Gold Chunk评测集，使用Agent同次调用产生的Retrieval Artifact计算Recall@3、Recall@5、MRR、Faithfulness和Context Recall；通过失败样本审计与Reranker消融将正式Full的 **Recall@5由76%提升至88%、MRR由0.4953提升至0.8067**，最终达到 **Doc-Hit@5 100%、Faithfulness 0.9382、Context Recall 0.9650**，50/50题有效、空回答0、Rerank降级0次。
- 使用 **FastAPI lifespan** 统一管理Milvus、VectorStore、Embedding和Agent资源，消除模块导入阶段连接外部服务的副作用；实现PDF异步任务状态持久化、中断识别、知识库异常503降级和MD/TXT原子索引切换，并增加上传大小、并发、CORS、Markdown清洗与脱敏日志等服务端安全边界。

## 四、面试展开要点

### 1. 为什么使用Agent而不是固定RAG流水线

用户既可能询问科研问题，也可能上传PDF、要求入库或查询任务状态。Agent负责识别意图和选择工具，具体检索与入库仍由确定性的后端服务执行，从而将模型决策和业务状态管理分离。

### 2. Prompt Engineering解决了什么

- 要求专业问题调用知识检索工具；
- 引用只能使用工具生成的作者—年份标签；
- 证据不足时必须明确说明；
- 区分病因、危险因素、相关因素、机制、检测方法和预测模型；
- 只有直接比较研究才能支持影响排序；
- 只有PDF状态为`indexed`时才能声称已经入库。

### 3. Context Engineering解决了什么

- Dense Top-20扩大初始候选覆盖；
- 精确去重避免相同Chunk挤占候选；
- 专用Reranker把直接证据前移；
- `source_id + chunk_index`补充同文献邻接证据；
- 12,000字符预算同时计算证据、分隔符和参考文献；
- Retrieval Artifact保存模型实际看到的完整上下文，用于前端展示和离线评测。

### 4. 如何定位性能瓶颈

先审计旧Full的12道Recall@5失败题，再对比原问题、Agent query、双路RRF和不同Reranker。结果发现部分正确Gold已经进入Dense候选，却被旧Listwise Rerank淘汰，因此主要瓶颈是重排稳定性，而不是继续扩大召回数量。

### 5. 为什么没有上线双路RRF

双路RRF消融的Recall@5达到92%，但生产知识工具当前只接收Agent query。稳定引入原始用户问题需要跨API、Agent和工具层传递额外状态。为控制线上复杂度，最终先落地生产同构的单路Agent query与专用Reranker；双路RRF只作为实验结论保留。

### 6. 后端可靠性如何保证

- 外部资源只在FastAPI lifespan中初始化和释放；
- Milvus初始化失败时服务可降级启动，相关入口返回503；
- PDF解析采用后台任务和持久化状态机；
- 服务重启后未完成任务标记为`interrupted`；
- MD/TXT更新采用写新版本、校验、再删除旧版本的原子切换；
- Reranker故障整批回退到向量Top-10；
- 原始评测结果、失败样本和run-specific目录全部保留。

## 五、真实性边界

- MCP与PubMed MCP尚未接入生产Agent，不能写成已实现能力。
- 双路Query/RRF、Query Rewrite、Multi-query和Hybrid Search尚未进入生产。
- 当前服务未实现用户认证、会话授权和费用配额，不能声称可直接公网部署。
- 不把检索消融的90%或92%写成正式Full成绩；正式Full Recall@5为88%。
- 不将Strict-Chunk-Hit解释为答案正确率。
- 不声称所有Chunk最大1,600字符；递归二次分割目标上限约3,200字符。
