## 2. 项目概述

系统面向AVF科研文献，提供文献上传、PDF解析、向量索引、Agent工具调用、两阶段检索、引用回答、SSE流式输出和全链路评测。

```text
文献侧：PDF/MD/TXT → Markdown/文本 → 分块 → Embedding → Milvus
问答侧：用户问题 → Agent → 检索工具 → 两阶段检索 → 证据上下文 → 回答
评测侧：证据优先题集 → 真实Chunk映射 → Full/BL-1 → ID指标与Ragas
```

## 5. Agent架构

`RagAgentService`负责：

- 初始化ChatQwen；
- 注册本地工具；
- 使用 `MemorySaver` 管理基于 `thread_id/session_id` 的进程内会话；
- 提供普通查询、带Trace查询和流式查询；
- 过滤内部Rerank模型输出，避免评分数字进入用户流；
- 对科研因果、排序和引用施加系统提示约束。

当前本地工具：

| 工具 | 作用 |
|---|---|
| 知识检索 | 执行完整检索链路，返回上下文和Artifact |
| 待处理PDF列表 | 返回尚未完成入库的PDF |
| PDF入库 | 提交后台解析和索引任务 |
| PDF状态查询 | 根据job_id查询进度 |
| 时间查询 | 返回时间信息 |

MCP接口已预留，但当前未配置外部服务。

## 6. 文档入库

### 6.1 MD/TXT

`POST /api/upload` 对MD/TXT执行：

1. 校验扩展名、大小和空内容；
2. 保存到 `uploads/`；
3. 删除相同来源的旧索引；
4. 分块；
5. 写入稳定Chunk元数据；
6. 生成Embedding并写入Milvus；
7. 区分上传成功与索引成功。

如果上传成功但索引失败，HTTP状态为207，响应体中的 `message` 为 `partial_success`。当前响应体 `code` 仍为200，属于待统一语义的问题。

### 6.2 PDF

PDF上传只登记原文：

```text
POST /api/upload
→ 校验 %PDF- 文件头
→ sha256
→ document_id = doc_{sha256[:6]}
→ uploads/originals/{document_id}/{filename}
→ 返回201、uploaded、indexed=false
```

只有用户明确要求解析或入库后，Agent才调用PDF入库工具。后台任务状态：

```text
queued → parsing → parsed → splitting → embedding → indexed
                                  ↘ failed
服务中断：运行中状态 → interrupted
```

解析结果写入 `uploads/parsed/{document_id}/`，任务状态写入 `uploads/jobs/{job_id}.json`。

## 7. 分块与索引

Markdown先按 `#`、`##` 标题分割，再使用递归字符分割器；小于300字符的块尝试合并。

当前事实：

```text
config.chunk_max_size = 1600
RecursiveCharacterTextSplitter.chunk_size = 1600 * 2 = 3200
chunk_overlap = 200
min_chunk_size = 300
```

因此“最大Chunk为1600字符”不是当前实现事实。小块合并条件也可能使最终Chunk边界略复杂，评测应统计实际Chunk长度。

新索引为每个Chunk写入：

```python
content_hash = sha256(content)[:16]
chunk_id = f"{document_id}:{content_hash}"
chunk_index = 文档内顺序
source_id = document_id或逻辑来源
```

这些字段支持精确去重、真实命中评测和邻居扩展。旧索引没有完整字段时，运行时会回退构造稳定逻辑ID，但无法恢复可靠文档内顺序。

## 8. 两阶段检索

### 8.1 默认流程

```text
Recall Top-50
→ 精确Chunk去重
→ LLM Rerank Top-20
→ rerank_score >= 0.65
→ 来源多样性选择（默认8个、每来源最多2个）
→ Top-3高分证据±1邻居扩展
→ ContextBuilder
```

### 8.2 精确去重

优先级：

```text
chunk_id → content_hash → 完整正文
```

该步骤只删除逻辑重复Chunk，不按论文来源删除不同证据。

### 8.3 来源多样性

来源多样性发生在Rerank和阈值过滤后，限制同一来源占据过多最终位置。它与候选精确去重目标不同。

### 8.4 相邻扩展

对最终高分证据尝试寻找同来源、`chunk_index ± 1` 的候选。只有新索引或已重建索引具有可靠Chunk顺序；历史数据的回退索引不能保证真实相邻关系。

### 8.5 上下文预算

总预算默认12000字符。ContextBuilder按分数排序：

1. 完整证据能放入则保留完整正文；
2. 不能放入时尝试按最多1600字符截断；
3. 仍无法放入则跳过该候选并继续后续候选。

所以1600是预算不足时的截断上限，不是所有证据的硬上限。

## 9. 检索Artifact

Artifact用于评测、日志和前端复核，核心字段：

```text
original_query
search_mode
candidate_count
reranked_count
selected_count
confidence
rerank_applied
documents[]:
  chunk_id
  source_id
  chunk_index
  content_hash
  source
  content
  vector_score
  rerank_score
  metadata
```

`query_with_trace()`确保评测读取的Artifact来自生成该答案的同一次Agent调用。

## 10. API与SSE

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/health` | 健康检查 |
| `POST` | `/api/chat` | 普通Agent问答 |
| `POST` | `/api/chat_stream` | SSE问答 |
| `POST` | `/api/chat/clear` | 清空会话 |
| `GET` | `/api/chat/session/{session_id}` | 会话历史 |
| `POST` | `/api/upload` | PDF/MD/TXT上传 |
| `POST` | `/api/index_directory` | 目录MD/TXT索引 |

主要SSE消息类型：

```text
tool_start
tool_call
retrieval_complete
search_results
content
done
error
```

Agent流式服务会过滤带 `internal_rerank` 标签的内部模型消息，只输出主模型节点的文本块，避免Rerank评分泄漏为用户可见数字。

## 11. 评测架构

### Legacy

25题、93条论文级标注，保留旧检索和去重消融结果。

### Full

50题、50篇独立文献，先映射真实Milvus逻辑Chunk ID，再通过 `query_with_trace()` 评估Agent实际检索上下文。

当前有效结果：Doc-Hit 49/50、Doc-Hit@1 82%、Doc-Hit@3 94%、Doc-Hit@5 98%、平均排名1.3。

### BL-1

Dense Top-5唯一Chunk，不启用Rerank、阈值、多样性和邻居扩展。已完成相同50题：Doc-Hit 94%、Doc-Hit@1 60%、Doc-Hit@3 88%、Doc-Hit@5 94%、平均排名1.6、Strict-Chunk-Hit 56%。

Full相对BL-1将Doc-Hit@1提升22个百分点、Doc-Hit@5提升4个百分点，平均排名由1.6改善至1.3；但Strict-Chunk-Hit由56%下降到36%。文档级提升可以报告，回答层和多可接受Chunk指标仍需补齐。

### 已知评测问题

- Full 50题Ragas均值被NaN污染；
- 详情没有保存完整回答、完整上下文和每题Ragas分数；
- Strict-Chunk-Hit只认可唯一目标Chunk，不能当作答案准确率。

## 15. 主要风险

1. 固定0.65阈值可能将所有候选过滤；
2. 分批Listwise Rerank分数跨批可比性有限；
3. Reranker输入截断可能遗漏超长Chunk后部证据；
4. 当前实际Chunk尺寸大于配置名直觉；
5. 旧索引缺少真实Chunk顺序；
6. Milvus历史数据存在重复UUID行；
7. MD/TXT覆盖与重新索引不是事务性操作；
8. `MemorySaver` 只在当前进程存活，会话不会跨重启持久化。
