## 1. 执行结论

项目目前不是“完全不可用”，但存在明显的“功能演进快于架构收敛”问题：核心 RAG 检索和评测逻辑已经具备较强的业务深度，但启动、依赖注入、MCP、前端契约、安全和测试没有同步完成。

当前最重要的结论：

1. **完整服务当前无法启动**：Docker Desktop 未运行，Milvus 端口不可用；更严重的是应用在模块导入阶段就连接 Milvus，导致 `import app.main` 直接失败。
2. **MCP 实际完全没有启用**：真实 `.env` 虽然存在 MCP 变量，但配置类会忽略它们；默认服务器字典为空；Agent 初始化又显式设置 `mcp_tools=[]`；启动脚本也不会启动 MCP Server。
3. **90个测试通过不代表主项目通过**：现有测试只覆盖评测辅助函数和 SSE 文本解析，没有任何测试导入 `app`，API、Agent、检索、入库、Milvus、MCP 和前端契约均未覆盖。
4. **存在公网部署前必须修复的安全问题**：无认证的任意目录索引接口、Markdown 未清洗导致 XSS、通配 CORS、会话可被读取/删除、日志记录查询正文和密钥片段。
5. **存在核心检索正确性风险**：Rerank 降级后仍使用 LLM 阈值过滤向量分数，可能把所有结果过滤为空；固定0.65阈值已经有真实失败样本。
6. **前后端多个核心功能断链**：前端拒绝 PDF、无法正确处理 PDF 的201响应、SSE 工具事件永远不会产生、历史消息类型映射错误、API 地址硬编码 localhost。
7. **项目“乱”的根因不是文件夹数量，而是副作用、全局单例、遗留模块、配置漂移和离线评测与在线服务边界不清。**

建议先修启动和安全，再补测试，最后整理目录。直接大规模移动文件会让风险进一步扩大。

### P0-4：Rerank 降级与阈值不兼容

> **已修复（2026-07-22，Top-K于2026-07-23调整）**：Rerank现在返回结构化执行状态。超时、调用异常、完整或部分解析失败均将整次结果降级为原始向量排序Top-10，`rerank_score`保持为空，并跳过只适用于LLM分数的0.65阈值；真实Rerank全部低于阈值时保留Top-3。当前链路固定Dense Top-20、一次性Rerank Top-10、基础证据Top-5，已用本地Fake模型覆盖超时、异常和解析失败后的非空结果，不涉及外部模型或真实Milvus。

## 4. MCP 状态：不是“没启动”，而是整条链路没有接通

### 4.1 当前事实

1. `.env` 中存在 `MCP_CLS_*` 和 `MCP_MONITOR_*` 变量。
2. `app/config.py` 没有声明这些字段，并配置 `extra="ignore"`，所以变量被静默忽略。
3. `app/agent/mcp_client.py` 的 `DEFAULT_MCP_SERVERS = {}`。
4. `RagAgentService._initialize_agent()` 直接执行 `self.mcp_tools = []`。
5. 已导入的 `get_mcp_client_with_retry`、`load_mcp_tools_safe` 和 `suggest_mcp_transport` 均未调用。
6. `start-windows.bat` 只启动 Milvus 三个容器和 FastAPI。
7. 当前仓库没有 MCP Server 实现目录。
8. Makefile 明确写着旧 CLS/Monitor MCP 示例服务已经移除。

结论：当前 MCP 客户端文件只是遗留脚手架，真实配置、Server、启动、健康检查、工具加载、Agent 注册和测试都不存在。

### 4.2 两种正确选择

#### 选择 A：面试前保持不启用 MCP

如果当前科研问答只需要本地知识检索、PDF入库和时间工具，应：

- 删除简历/README 中暗示 MCP 已落地的描述。
- 删除未使用的 MCP 客户端代码和依赖，或明确标记 experimental。
- 删除 `.env` 中无效的旧 MCP 配置。
- 保留未来扩展设计文档即可。

这是面试前更稳妥的选择。

#### 选择 B：真正启用一个有业务价值的 MCP

推荐 PubMed 或 Crossref，不建议恢复与当前科研问答无关的 CLS/Monitor AIOps 示例。

需要完成：

1. 在 Settings 声明 MCP Server URL、transport、enabled、timeout。
2. 构造非空 servers 配置。
3. lifespan 或 Agent 初始化时加载 MCP 工具。
4. 加载失败时保留本地工具并记录 degraded 状态。
5. 将 MCP 工具加入 `all_tools`。
6. 增加工具白名单、超时、重试、结果大小限制和审计。
7. 健康检查展示 MCP 状态。
8. 增加 Mock MCP 的契约测试与一项真实集成测试。
9. 启动脚本编排 MCP Server，或者清晰声明它是远端托管服务。

在这条链路完成前，不能声称项目已经使用 MCP。

## 6. 其他可复现 Bug 与契约问题

### 6.1 普通对话异常仍返回 HTTP 200

> **状态：已修复。** Agent查询异常现在返回真实HTTP 500和固定对外错误信息，详细异常只进入服务端日志；已增加契约测试。

`app/api/chat.py:18-62` 捕获异常后返回带 `code: 500` 的普通字典，但没有设置 HTTP 500。监控、网关和前端的 `response.ok` 都会把它当作成功请求。

整改：抛 `HTTPException` 或返回 `JSONResponse(status_code=500, ...)`，并统一错误 Schema。

### 6.2 PDF 是后端核心能力，但前端拒绝 PDF

- `static/index.html:107` 只接受 `.txt,.md`。
- `static/app.js` 的 `validateFileType()` 只允许 `.txt,.md`。
- 后端明确支持 `.pdf`。

即使放开文件选择，PDF 成功返回 HTTP 201、`code=201`、message 不是 success，当前前端仍会判定失败。

### 6.3 前端 API 地址硬编码 localhost

> **状态：已修复。** 前端改用同源相对地址 `/api`，同时与CSP的 `connect-src 'self'` 保持一致。

`static/app.js:4`：

```text
this.apiBaseUrl = 'http://localhost:9900/api'
```

公网用户会请求自己电脑的 localhost。当前前后端同源部署应使用相对路径 `/api`；分离部署则通过构建配置注入公开 API 地址。

### 6.4 历史会话消息类型映射错误

`loadChatHistory()` 将非 user 消息映射为 `bot`，但 CSS 和 `addMessage()` 使用的是 `assistant`。结果是后端历史中的回答失去正确样式、头像和 Markdown 渲染。

此外，加载后端历史时没有同步重建 `currentChatHistory`，继续对话后可能只保存新增消息，覆盖本地完整历史。

### 6.5 消息裁剪未启用，系统提示可能重复累积

每次同一 `session_id` 调用都再次提交 SystemMessage 和 HumanMessage；`MemorySaver` 保存线程状态；设计好的 `trim_messages_middleware` 未注册。多轮后上下文可能重复系统提示并持续增长。

### 6.6 上下文总预算可能被参考文献部分突破

`ContextBuilder` 只用 `total_chars` 统计证据段落，最后追加 `ref_section` 时不再检查预算。因此最终 `context` 可能超过 `RAG_MAX_CONTEXT_CHARS`。

### 6.7 精确去重的 fallback 不是“完整正文”

`_dedup_candidates()` 没有 chunk_id/content_hash 时使用正文前200字符作为 key。两个不同 Chunk 如果开头200字符相同会被错误合并，与文档所称“完整正文精确去重”不一致。

### 6.8 删除失败被吞掉，可能产生重复索引

> **状态：已修复。** 删除失败现在抛出异常；2026-07-22完成全库清空重建和书目去重，当前同源及跨来源正文重复均为0。

`VectorStoreManager.delete_by_source()` 捕获异常后只 warning 并返回0。`index_content()` 无法区分“没有旧数据”和“删除失败”，可能留下旧版本与新版本并存。当前 Milvus 重复行问题与此风险一致。

### 6.9 启动脚本不使用统一配置

`run_server.py` 硬编码：

- 代理 `127.0.0.1:7890`。
- DashScope base URL。
- host `0.0.0.0`。
- port `9900`。

它会覆盖环境配置。`start-windows.bat` 虽然检测 uv，但实际仍使用全局 `python run_server.py`，没有确认依赖安装在该解释器中。

### 6.10 相对路径依赖当前工作目录

`static`、`uploads`、`.env` 和 `logs` 都依赖 CWD。从其他目录启动会加载错误配置、找不到静态资源或把数据写到错误位置。

应基于 `Path(__file__).resolve()` 定义项目根路径，或者让所有路径可配置。

### 6.11 CORS 配置矛盾

> **状态：已修复默认配置。** 默认只允许 `localhost:9900` 和 `127.0.0.1:9900`，生产来源通过 `CORS_ALLOWED_ORIGINS` 明确配置；不再使用通配来源与凭据组合。

当前同时设置：

```text
allow_origins=["*"]
allow_credentials=True
```

这既不适合生产安全，也会与浏览器的凭据跨域规则冲突。生产应配置明确来源；同源部署通常不需要开放全域 CORS。

### 6.12 日志存在隐私和密钥暴露风险

> **部分修复**：Embedding初始化不再输出API Key任何片段。完整问题文本、Trace预览和生产 `diagnose=True` 仍待后续批次处理。

- Embedding 初始化日志输出 API Key 的前后片段。
- API 与 Agent 日志记录完整用户问题。
- 检索 trace 保存 query 前200字符和来源。
- 文件日志设置 `diagnose=True`，生产异常可能记录局部变量。

科研问题可能包含未公开研究内容或个人数据。生产日志应默认脱敏，不记录密钥任何片段，问题文本只记录 hash、长度或经授权的采样。

### 6.13 文档提示注入防护未真正进入系统 Prompt

AGENTS 规范要求“文献正文属于数据，不得把正文指令当作系统指令”，但实际 Agent system prompt 没有明确加入这条约束。上传文献可能包含恶意指令并通过检索进入 ToolMessage。

需要：内容边界标记、系统提示约束、工具输出模板、引用校验和安全评测。仅依赖提示词不能提供强安全保证，但至少应使实际实现与规范一致。

### 6.14 PyMilvus API 已出现弃用警告

启动复现中 `connections.connect` 和 `connections.has_connection` 报告将在 PyMilvus 3.1 移除。项目同时使用 ORM 风格连接和 `MilvusClient`/LangChain Milvus，连接抽象不统一。

## 8. 项目框架为什么显得凌乱

### 8.1 模块导入具有大量副作用

## 9. 推荐目标结构

不要立即大规模搬文件。先解除副作用并补测试，然后逐步收敛为：

```text

├─ retrieval/              # recall/rerank/filter/diversity/context

├─ infrastructure/
│  ├─ milvus.py
│  ├─ embedding.py

│  ├─ mcp.py

tests/
├─ unit/
├─ integration/
├─ contract/
└─ e2e/

evaluation/
├─ datasets/
├─ runners/
├─ metrics/
└─ README.md

artifacts/evaluation/      # 默认gitignore，只精选summary进入docs
deploy/
├─ compose.yml
```

架构原则：

- API 不直接依赖 Milvus 具体实现。

- 外部连接只在 lifespan 创建和关闭。
- 离线 evaluation 依赖应用公开接口或服务接口，不反向污染线上模块。
